## Part 1 — Color Spaces: What Each Is Good For

### 1.1 sRGB — the lingua franca, perceptually broken

sRGB (IEC 61966-2-1, 1996) is the universal device-encoding space for the web, Windows GDI/Direct2D/SharpDX, and almost every chart library. It is **gamma-encoded**: stored as `R'G'B'` where the prime indicates a non-linear transfer function applied to compress brightness into 8 bits more efficiently than linear light would.

Two facts matter for the agent:

1. **You cannot interpolate in sRGB without producing muddy intermediates.** A 50% blend of `#ff0000` (red) and `#00ff00` (green) in raw 8-bit sRGB gives `#808000`, which is a dirty olive — *not* the bright yellow your eye expects. The reason: sRGB is gamma-encoded, so adding two encoded values doesn't add their physical light.
2. **Equal RGB steps are not equal perceptual steps.** Going from R=10 to R=20 looks like a huge change; going from R=200 to R=210 is invisible. This is the entire reason perceptually uniform spaces exist.

**Use sRGB for:** final output to the framebuffer, hex code storage, file format I/O. **Never for:** interpolation, palette generation, distance calculations, contrast math.

### 1.2 Linear sRGB — the missing intermediate

Linear sRGB is what you get when you undo the gamma curve:

```
to_linear(c):
    c_n = c / 255
    if c_n <= 0.04045:
        return c_n / 12.92
    else:
        return ((c_n + 0.055) / 1.055) ^ 2.4
```

This is what actually behaves like physical light. It is also what WCAG 2 contrast and APCA both consume internally. But it is *not* perceptually uniform either — it is *radiometrically* uniform.

### 1.3 HSL / HSV — fast, popular, perceptually wrong

HSL (Hue, Saturation, Lightness) and HSV (Hue, Saturation, Value) are simple geometric reshapings of sRGB. They feel intuitive — you can adjust "hue" and "lightness" independently.

The catastrophe: **HSL Lightness is not perceptual lightness.** `hsl(60, 100%, 50%)` is pure yellow `#ffff00` — visually one of the brightest colors a sRGB display can produce. `hsl(240, 100%, 50%)` is pure blue `#0000ff` — visually much darker. Both are reported as 50% lightness. A palette generated by sweeping HSL hue at fixed L will pulse brighter and darker as you rotate.

**Use HSL/HSV for:** quick mockups, color picker UIs, and *only* when perceptual uniformity does not matter. **Never** as the basis for a trading-system palette.

### 1.4 CIELAB / CIELCH — the 1976 perceptual standard

CIELAB (CIE L\*a\*b\*, 1976) was the first widely adopted perceptually uniform space. L\* is perceptual lightness (0 to 100), a\* is green↔red, b\* is blue↔yellow. Its cylindrical form CIELCH replaces a\*, b\* with C\* (chroma) and h° (hue angle).

CIELAB has a known **blue-shift problem**: as you increase chroma in the blue region, the hue drifts toward purple (Ottosson's central critique). Gradients through blue look wrong. It also overweights yellow lightness slightly.

**Use CIELCH for:** quality-control color matching, ΔE2000 comparisons, scientific publication colormaps. **Use OKLCH instead for:** UI palettes, gradients, palette derivation. Modern.

### 1.5 OKLCH / Oklab — the modern standard (2020)

Björn Ottosson's Oklab (December 2020) is fitted to CAM16 lightness/chroma data and IPT hue data. It is the recommended space in CSS Color Level 4, used by **Tailwind v4** (default palette), **Radix UI** (palette generation in OKLCH), **Shopify Polaris**, and the **Lightroom / Linear / Vercel** design teams. Browser support has been universal since 2023.

The cylindrical Oklch form (often written `oklch()` in CSS) is the operational format:

- **L** in [0, 1] (sometimes shown as 0–100%): perceived lightness. L=0.5 is mid-gray for *every* hue.
- **C** in [0, ~0.4 in practice]: chroma. C=0 is gray; max C depends on hue and L.
- **H** in [0°, 360°): hue angle. ~30° = orange, 145° = green, 264° = blue, 350° = red.

**Use OKLCH for:** every palette decision in DEEP6.

### 1.6 CIECAM02 / CAM16 — the most accurate, the most complex

CIECAM02 (2002, superseded by CAM16 in 2016) is a full Color Appearance Model. It accounts for the viewing environment (background luminance, surround, adaptation state) and produces six attributes: brightness, lightness, colorfulness, chroma, saturation, hue.

CAM16-UCS is the uniform color space derived from CAM16 — used by Material 3 (HCT is essentially a CAM16 derivative) and by Adobe Leonardo internally.

**Use CAM16/HCT for:** academic projects, video pipelines, theoretical perfection. **Use OKLCH for:** everything else. OKLCH is a lightweight approximation of CAM16 that runs in 12 lines of math; CAM16 is hundreds of lines and depends on viewing-condition assumptions you cannot guarantee for a NinjaTrader chart panel.

### 1.7 Comparison Table

| Space | Perceptually uniform? | Hue stability | Cost | Use it for |
|---|---|---|---|---|
| sRGB (gamma) | No | N/A | Free | Storage, output |
| Linear sRGB | No (radiometric) | N/A | 1 pow | WCAG luminance |
| HSL / HSV | **No** | Bad | Free | Color pickers only |
| CIELAB / CIELCH | OK | Blue shifts | 1 cube root + matrix | Print, ΔE matching |
| **OKLCH** | **Yes** | **Excellent** | 1 cube root + matrix | **Palette gen, gradients, DEEP6** |
| CIECAM02 / CAM16 | Yes | Excellent | Heavy | Research, dynamic theming |

## Part 2 — OKLCH Deep-Dive

### 2.1 Forward conversion: linear sRGB → Oklab

The hot path for the agent is **linear sRGB → Oklab**, skipping CIE XYZ. Ottosson published the fused matrices.

**Step 1 — Linear sRGB to LMS cone responses** (matrix M₁):

```
l = 0.4122214708·R + 0.5363325363·G + 0.0514459929·B
m = 0.2119034982·R + 0.6806995451·G + 0.1073969566·B
s = 0.0883024619·R + 0.2817188376·G + 0.6299787005·B
```

**Step 2 — Cube-root non-linearity:**

```
l' = cbrt(l)
m' = cbrt(m)
s' = cbrt(s)
```

**Step 3 — Non-linear LMS to Lab** (matrix M₂):

```
L = 0.2104542553·l' + 0.7936177850·m' − 0.0040720468·s'
a = 1.9779984951·l' − 2.4285922050·m' + 0.4505937099·s'
b = 0.0259040371·l' + 0.7827717662·m' − 0.8086757660·s'
```

### 2.2 Cylindrical conversion: Oklab → OKLCH

```
C = sqrt(a² + b²)
h = atan2(b, a)            // radians; convert to degrees: × 180/π
```

Inverse:

```
a = C · cos(h_radians)
b = C · sin(h_radians)
```

### 2.3 Inverse: Oklab → linear sRGB

Apply M₂⁻¹, cube the components, apply M₁⁻¹. The inverse matrices (Ottosson, public domain):

**M₂⁻¹ (Lab → l'm's'):**

```
l' = L + 0.3963377774·a + 0.2158037573·b
m' = L − 0.1055613458·a − 0.0638541728·b
s' = L − 0.0894841775·a − 1.2914855480·b
```

Then `l = l'^3`, `m = m'^3`, `s = s'^3`, and:

**M₁⁻¹ (LMS → linear RGB):**

```
R = +4.0767416621·l − 3.3077115913·m + 0.2309699292·s
G = −1.2684380046·l + 2.6097574011·m − 0.3413193965·s
B = −0.0041960863·l − 0.7034186147·m + 1.7076147010·s
```

### 2.4 Lightness preservation across hues (the key property)

In OKLCH, a green at `oklch(0.65 0.18 145)` and a red at `oklch(0.65 0.18 25)` look like the same lightness. In HSL, `hsl(120 100% 50%)` is dramatically brighter than `hsl(0 100% 50%)`. This is *the* property that lets you build a complete palette by walking hue at a fixed L.

**Rule for the agent:** When you need 8 distinguishable categorical colors that look balanced (no one pulling rank), pick a single L in [0.65, 0.75] and a single C in [0.13, 0.18] and rotate H in equal angular steps.

### 2.5 Browser support and fallback

`oklch()` and `oklab()` ship in Chromium 111+ (March 2023), Safari 15.4, Firefox 113. For NinjaTrader (.NET WPF / Windows Forms / SharpDX), **there is no native support** — you compute everything offline and emit sRGB hex / `Color.FromArgb(...)`.

Web fallback pattern:

```css
:root {
  --accent: #f97316;                    /* sRGB fallback */
  --accent: oklch(0.70 0.18 38);        /* OKLCH primary */
}
```

### 2.6 Gamut mapping (the hard part)

Many OKLCH triples are out of sRGB gamut. `oklch(0.55 0.32 264)` is a more saturated blue than sRGB can show. The agent must **gamut-map** before emitting hex.

The Culori / CSS Color 4 algorithm: hold L and H constant, **bisect on C** down from the requested chroma until the result is inside sRGB. This preserves lightness and hue at the cost of saturation — the perceptually correct tradeoff.

```
gamut_map(L, C_target, H):
    if oklch_to_srgb(L, C_target, H) is in [0,1]³: return that
    lo, hi = 0, C_target
    for _ in 16:                           // 16 iterations = chroma resolved to ~5e-6
        mid = (lo + hi) / 2
        rgb = oklch_to_srgb(L, mid, H)
        if rgb in [0,1]³: lo = mid
        else:              hi = mid
    return oklch_to_srgb(L, lo, H)
```

## Part 3 — Building a Complete Color System from One Accent

### 3.1 The 11-step Tailwind scale

Tailwind v4's default palette uses **11 stops** per hue: 50, 100, 200, 300, 400, 500, 600, 700, 800, 900, 950. The default Tailwind L values (Evil Martians' analysis):

```
L = [0.9778, 0.9356, 0.8811, 0.8267, 0.7422,
     0.6478, 0.5733, 0.4689, 0.3944, 0.3200, 0.2378]
```

These are *not* algorithmic — they were hand-picked. But once you fix them, every hue can be slotted in at the same L.

A safe **fixed-chroma curve** (max-consistency mode from Tailwind v4, applies across all hues):

```
C = [0.0108, 0.0321, 0.0609, 0.0908, 0.1398,
     0.1472, 0.1299, 0.1067, 0.0898, 0.0726, 0.0540]
```

The peak chroma is at step 500 (~0.147) where the eye expects the "main" color of a hue, then chroma falls off in both directions.

### 3.2 Generating a scale from one accent

Given a single accent in OKLCH `(L_acc, C_acc, H)`, derive 11 stops:

```python
def scale_from_accent(L_acc, C_acc, H):
    L = [0.9778, 0.9356, 0.8811, 0.8267, 0.7422,
         0.6478, 0.5733, 0.4689, 0.3944, 0.3200, 0.2378]
    # Anchor chroma so accent's chroma lands at the closest step
    base_C = [0.0108, 0.0321, 0.0609, 0.0908, 0.1398,
              0.1472, 0.1299, 0.1067, 0.0898, 0.0726, 0.0540]
    scale_factor = C_acc / max(base_C)        # normalize to user's saturation taste
    C = [c * scale_factor for c in base_C]
    return [(L[i], C[i], H) for i in range(11)]
```

For a **monochromatic** scale that holds the accent hue exactly: just hold H constant. For a **harmonious** secondary palette, rotate H by +180° (complementary), +120° (triadic), +60° (analogous near), +30° (analogous tight).

### 3.3 Material 3 HCT vs Tailwind OKLCH

Material 3 uses **HCT** (Hue, Chroma, Tone — Tone is CIE-style L\* approximated via CAM16). HCT generates **5 tonal palettes** from one source: Primary, Secondary, Tertiary, Neutral, Neutral-Variant. Then 13 tones (0, 10, 20, ..., 100) per palette.

The mapping to Material color roles uses the table:

```
Primary (light theme)         = Primary palette tone 40
On Primary (light theme)      = Primary palette tone 100
Primary (dark theme)          = Primary palette tone 80
On Primary (dark theme)       = Primary palette tone 20
Surface (light)               = Neutral palette tone 99
Surface (dark)                = Neutral palette tone 10
```

For DEEP6 use OKLCH (lighter, simpler, no viewing-condition assumptions). HCT is overkill for a fixed-context trading dashboard.

### 3.4 When to break the system

Some semantic colors **must not** come from your accent rotation:

- **`buy`** / `gain` — always near OKLCH H≈145° (green).
- **`sell`** / `loss` — always near OKLCH H≈25° (red).
- **`warning`** — always near OKLCH H≈75° (yellow/amber).
- **`danger`** — H≈25° (red), but with higher chroma than `loss`.

These map to physiological/cultural expectations and are immune to the accent. Allow only L and C tweaks per theme; never the hue.

## Part 4 — Perceptually Uniform Diverging Gradients (Delta Heatmaps)

### 4.1 Why linear RGB interpolation fails

A red→blue gradient interpolated in sRGB passes through `#808080` mid (mud). In linear sRGB it passes through a too-dark gray. In OKLCH (with L held high through the middle) it passes through a clean light gray or pale neutral, with no visible "dead zone."

**Empirical test:** generate red→blue at 21 stops in each space, view as a band, count how many distinct steps a viewer can see. sRGB interpolation: ~14. OKLCH (linear in L,C,H with hue going short way): ~21.

### 4.2 Bezier interpolation (chroma.js technique)

For a 3-color diverging gradient (red → white → blue), naive linear interpolation in Lab/OKLCH gives a sharp corner at the midpoint. Cubic Bezier interpolation in Lab smooths it: the endpoints anchor the curve, the midpoint is a control point that the curve is *guided by* but does not pass through. The result is a smoother gradient with the midpoint hue still recognizably neutral.

```js
// chroma-js
const scale = chroma.bezier(['#b2182b', '#f7f7f7', '#2166ac']).scale();
// equivalent in OKLCH: interpolate L,C,H independently with cubic Bezier, then convert
```

### 4.3 The matplotlib Perceptually Uniform colormaps

`viridis`, `inferno`, `magma`, `plasma`, `cividis` (van der Walt & Smith 2015; Nuñez/Anderton/Renslow 2018 for cividis). All five are designed in CAM02-UCS to be:

1. **Monotonically increasing in lightness** — so a B/W print preserves the order.
2. **Equal perceptual steps** — the eye's "more-than" judgment is reliable.
3. **Color-blind safe** for deuteranopia (and cividis is also tritanopia-safe).

For DEEP6 sequential heatmaps (e.g. order book volume by price level), **viridis** is the safe default. For a high-detail map (where you want to see fine differences), **inferno** has higher contrast slope. For absolute color-blind safety regardless of CVD type, **cividis**.

### 4.4 Turbo — when you want jet's pop without jet's lies

Google's Turbo (Mikhailov 2019) is a *replacement* for the universally-rejected `jet` rainbow. It is **not** perceptually uniform, but its lightness slope is monotonic at the ends and roughly double viridis's, making subtle structure visible. CVD-tested (works for protan/deutan/tritan, fails only for total achromatopsia ~0.003%).

Use Turbo only when chart density requires extreme contrast and your audience accepts the non-uniform spacing. For a DEEP6 footprint cell heatmap with 200+ cells visible at once, Turbo wins on detail extraction; viridis wins on quantitative accuracy.

### 4.5 Diverging palettes for delta visualization

The **delta** (buy volume − sell volume) is fundamentally diverging: 0 is the meaningful center, ±N are extremes. Use:

- **RdBu** (ColorBrewer) — red negative, blue positive. Politically/culturally neutral.
- **RdYlGn** — red→yellow→green. Familiar from finance, BUT the red↔green pair is **the worst** for deuteranopes. Avoid for trader UIs.
- **BrBG** — brown→white→teal. Color-blind safe and used by climate scientists.
- **Custom OKLCH-derived** — see Part 12 for DEEP6's recommended diverging palette.

### 4.6 Stop-count optimization

For a smooth gradient rendered as a CSS `linear-gradient`, **5 stops** is enough if interpolation is in OKLCH. **3 stops** suffices for diverging if Bezier-interpolated. A precomputed LUT for a heatmap should be **256 entries** (one per byte), generated offline in OKLCH and cached as sRGB hex.

## Part 5 — Color-Blind Safety Math

### 5.1 The three dichromacies

| Type | Cone missing | Prevalence (men) | Net effect |
|---|---|---|---|
| **Protanopia** | L (long, "red") | 1.0% | Red ↔ green ↔ brown collapse; reds dim |
| **Deuteranopia** | M (medium, "green") | 1.1% | Red ↔ green collapse; saturation reduced |
| **Tritanopia** | S (short, "blue") | 0.001% | Blue ↔ yellow collapse |

Plus **anomalous trichromacies** (deuteranomaly is most common at ~5% of men) where the cone is shifted, not missing — same axis collapse, less severe.

Total CVD prevalence: ~8% of men, ~0.5% of women. For a global trading platform, **assume one in every twelve users** is colorblind.

### 5.2 The Brettel-Viénot-Mollon algorithm

Brettel, Viénot, Mollon (JOSA A 1997) — the algorithm for simulating dichromat vision. The pipeline:

1. **sRGB → linear RGB → CIE XYZ → LMS** (Hunt-Pointer-Estevez or Smith-Pokorny matrix).
2. In LMS space, **project each pixel onto the dichromat's reduced surface**: a 2D plane defined by the neutral axis (white-to-black) and the residual hue axis (the hues a unilateral dichromat reports).
3. Plane is **piecewise** — split by a hue line — to handle the discontinuity correctly.
4. **LMS → XYZ → linear RGB → sRGB** to view.

This is the gold-standard simulation. The agent does NOT render in CVD-simulated colors for the user; it **uses the simulator to verify** that two emitted colors are still distinguishable.

### 5.3 The verification routine

For every pair of colors the agent emits as "must be distinguishable" (e.g. buy vs sell, signal A vs signal B):

```python
def cvd_safe(c1_hex, c2_hex, threshold_dE2000=10):
    for cvd_type in ['protan', 'deutan', 'tritan']:
        s1 = brettel_simulate(c1_hex, cvd_type)
        s2 = brettel_simulate(c2_hex, cvd_type)
        if delta_e_2000(s1, s2) < threshold_dE2000:
            return False, cvd_type
    return True, None
```

ΔE2000 ≥ 10 is "obviously different" for everyone. ΔE2000 ≥ 3 is "noticeable." ΔE2000 < 1 is "indistinguishable."

## Part 6 — WCAG 2.2 Contrast Math

### 6.1 sRGB relative luminance

```
L_rel(R, G, B) = 0.2126·R_lin + 0.7152·G_lin + 0.0722·B_lin
```

Where each channel is sRGB→linear via the formula in §1.2. The coefficients `0.2126, 0.7152, 0.0722` are the Y row of the sRGB→XYZ matrix (D65 white point).

### 6.2 Contrast ratio

```
contrast_ratio(L1, L2) = (L_lighter + 0.05) / (L_darker + 0.05)
```

Range 1:1 (identical) to 21:1 (pure black on pure white). The `+ 0.05` term is the "viewing flare" assumption — ambient light reflecting off the screen — without which white text on black would have infinite contrast.

### 6.3 WCAG 2.2 thresholds

| Threshold | Requirement | Use |
|---|---|---|
| 3:1 | AA large text (≥18pt or 14pt bold) | Headers, axis labels (large) |
| 4.5:1 | AA normal text | Body text, prices, P&L |
| 7:1 | AAA normal text | Critical numerics, alert text |
| 3:1 | AA non-text (UI components) | Borders, focus rings |

### 6.4 The pure-black trap

WCAG 2 contrast says white-on-pure-black is 21:1 — the maximum. But on an OLED, fast-moving white text on `#000000` exhibits *halation* — the iris dilates wide, and the bloom of bright pixels into the dead-pixel surround creates ghosting. Astigmatic users see white text "smear."

The fix: drop the background to ~`#0A0A0A` or `#0D1117`. Contrast falls from 21:1 to ~19.5:1 — still 4× over AAA — but halation is eliminated.

### 6.5 APCA — the proposed WCAG 3 replacement

APCA (Accessible Perceptual Contrast Algorithm, Andrew Somers) is the candidate for WCAG 3. Unlike WCAG 2's symmetric ratio, APCA produces an **Lc value** (lightness contrast) from 0 to ±106, where:

- **Sign matters:** positive Lc = dark text on light bg (BoW); negative Lc = light text on dark bg (WoB).
- **Doubling the number doubles perceived contrast** (unlike the 1.05/1.05 → 21/1.05 nonlinearity of WCAG 2).
- **Text size weighting:** APCA's full silver/gold tables relate Lc to font size & weight.

#### 6.5.1 The APCA-W3 constants (verified from `apca-w3` v0.1.x source)

```
mainTRC      = 2.4          // input gamma exponent
sRco         = 0.2126729
sGco         = 0.7151522
sBco         = 0.0721750

normBG       = 0.56         // exponent on bg luminance, BoW polarity
normTXT      = 0.57         // exponent on text luminance, BoW polarity
revTXT       = 0.62         // exponent on text luminance, WoB polarity
revBG        = 0.65         // exponent on bg luminance, WoB polarity

blkThrs      = 0.022        // soft black-clamp threshold
blkClmp      = 1.414        // soft black-clamp exponent

scaleBoW     = 1.14
scaleWoB     = 1.14
loBoWoffset  = 0.027
loWoBoffset  = 0.027
loClip       = 0.1
deltaYmin    = 0.0005
```

#### 6.5.2 The APCA Lc formula (BoW = dark text on light bg)

```
1. Y_bg   = sRco·R_bg^2.4 + sGco·G_bg^2.4 + sBco·B_bg^2.4    (R,G,B in [0,1])
2. Y_txt  = sRco·R_t^2.4  + sGco·G_t^2.4  + sBco·B_t^2.4

3. soft-clamp blacks:
   if Y < blkThrs: Y += (blkThrs - Y)^blkClmp

4. if |Y_bg - Y_txt| < deltaYmin: return 0

5. if Y_bg > Y_txt:                 // BoW polarity
       SAPC = (Y_bg^normBG - Y_txt^normTXT) * scaleBoW
       Lc   = (SAPC < loClip) ? 0 : (SAPC - loBoWoffset) * 100
   else:                            // WoB polarity
       SAPC = (Y_bg^revBG  - Y_txt^revTXT)  * scaleWoB
       Lc   = (SAPC > -loClip) ? 0 : (SAPC + loWoBoffset) * 100
```

#### 6.5.3 APCA Lc thresholds for DEEP6

| Lc | Use |
|---|---|
| ±90 | Preferred for body text, fluent reading |
| ±75 | Minimum for body text (≥18px) |
| ±60 | Minimum for content text (≥24px) |
| ±45 | Large text (≥36px), data labels |
| ±30 | Large/bold display text only |
| ±15 | Decorative non-text |

For a trading UI where readability under fatigue is paramount, target **Lc ≥ |75|** for every price, P&L, and signal value.

### 6.6 Why pure black + saturated color = visual vibration

The Helmholtz-Kohlrausch effect (Part 8) makes saturated colors appear brighter than they are. On `#000000`, a saturated cyan value like `#00FFFF` looks nuclear — eye fights to focus, edges shimmer. Drop bg to `#0D1117` and desaturate cyan to OKLCH(0.85, 0.12, 200), and the effect disappears.

## Part 7 — Background Luminance Science

### 7.1 OLED vs LCD — different failure modes

**OLED:** each pixel is its own light source. `#000000` = pixel completely off. Pros: infinite contrast, perfect blacks, power-saving. Cons:
- **Black smear** — pixels take ~5–15ms to wake from off; fast scrolling shows ghosting on dark→light transitions.
- **Burn-in** — static UI elements (price headers, watchlists) at high brightness etch permanent ghosts. Mitigation: shift UI position by 1–2px every 10 minutes ("pixel shift").

**LCD:** backlight is always on; pixels modulate transmitted light. `#000000` is never truly black (backlight bleed visible in dark room). Pros: no smear, no burn-in. Cons: imperfect blacks, halo/bloom around bright UI on dark bg, IPS glow at viewing angles.

### 7.2 Reference background recipes

| Brand / system | Bg hex | OKLCH | Why |
|---|---|---|---|
| Bloomberg Terminal | `#000000` | (0, 0, 0) | Amber accent (#FFA028) demands max contrast; trader expectation |
| GitHub dark | `#0D1117` | (0.180, 0.014, 254) | Slight blue tint for "cool" perception, off-pure-black for OLED safety |

### 7.3 The trader-specific rule for DEEP6

For a footprint chart with **dense data** (200+ cells, signals overlaid, axis labels, P&L panel):

- Background `oklch(0.16 0.005 240)` ≈ `#101418` — near-black with the faintest cool tint.
- Chart canvas `oklch(0.20 0.005 240)` ≈ `#181E24` — slightly elevated for the actual price/volume area.
- Panel surfaces `oklch(0.24 0.005 240)` ≈ `#222932` — for sidebar, trade ticket.

The **3-step elevation** (16 → 20 → 24 in OKLCH lightness) maps to Material's surface tint system without using saturated overlays.

## Part 8 — Saturation Discipline (the Science)

### 8.1 The Helmholtz-Kohlrausch effect

Discovered ~1860 (Helmholtz) and quantified by Kohlrausch (1923): **saturated colors appear brighter than less-saturated colors at the same luminance**.

The effect is strongest in **red-magenta** and **blue** hues, weakest in yellow. For a trading UI, this means:

- A small saturated red dot will appear *brighter* than a large neutral-gray rectangle of the same physical luminance.
- **Saturated colors fight for attention even when small.** Reserve them for events that genuinely demand attention (fills, alerts, signal triggers).

### 8.2 The three-tier color hierarchy

Drawing from scientific viz practice (Munsell, Tufte) and codified by IBM Carbon and Material 3:

```
Tier 1 — Background and chrome
    Neutral; OKLCH C ≤ 0.02; visually recedes
    Examples: panel backgrounds, dividers, axis lines

Tier 2 — Data and structure
    Low-to-medium saturation; OKLCH C in [0.05, 0.12]
    Examples: candle bodies, footprint cells, volume bars

Tier 3 — Signals and alerts
    High saturation; OKLCH C in [0.15, 0.22]
    Examples: signal arrows, fill markers, P&L change flashes
```

**Inverse rule:** the smaller and more transient the element, the more saturated it is allowed to be.

### 8.3 Munsell mapping

Munsell's `Hue Value/Chroma` notation (e.g., `5R 4/14`) is the conceptual ancestor of OKLCH. The three dimensions are independent in human perception. Munsell's lasting contribution: **the chroma scale is open-ended** — there is no upper bound, unlike HSL's misleading 100%.

For DEEP6: think in Munsell terms when sketching, in OKLCH when computing.

## Part 9 — Heatmap Color Choice

### 9.1 Three families

**Sequential** (single anchor, monotonic)
- Use when: data is ordered low→high (volume, price intensity, time-since-event).
- Default: `viridis` for accuracy, `inferno` for contrast.

**Diverging** (two anchors, neutral middle)
- Use when: data has a meaningful zero (delta, P&L, deviation from VWAP).
- Defaults: `RdBu` for politically neutral, `BrBG` for color-blind safe, custom OKLCH for brand-aligned.

**Categorical** (no order)
- Use when: distinguishing classes (signal types, account groups).
- Defaults: Tableau 10, Okabe-Ito (8 colors), IBM 5-color.

### 9.2 Why "rainbow" / jet is universally rejected

The jet colormap (MATLAB legacy, blue-cyan-green-yellow-red) is **discontinuous in lightness** — it goes dark, light, dark, light, dark. Visual artifacts:

- Yellow band appears as a sharp ridge of light, regardless of whether the data has one.
- Cyan-green transition is invisible to deuteranopes.
- Hue-only mapping creates illusory "edges" at perceptual boundaries that don't correspond to data boundaries.

Rothman, Borland, & Taylor (2007) and Crameri et al. (2020) document the harm: scientific papers using jet have measurably higher misinterpretation rates.

**Hard rule for the agent:** *never* generate a jet/rainbow colormap. Substitute Turbo if the brief is "high-contrast rainbow."

### 9.3 Categorical palette catalog

**Tableau 10** (Maureen Stone, Tableau 10 release, 2016):

| # | Hex | Name |
|---|---|---|
| 1 | `#5778A4` | Blue |
| 2 | `#E49444` | Orange |
| 3 | `#D1615D` | Red |
| 4 | `#85B6B2` | Teal |
| 5 | `#6A9F58` | Green |
| 6 | `#E7CA60` | Yellow |
| 7 | `#A87C9F` | Purple |
| 8 | `#F1A2A9` | Pink |
| 9 | `#967662` | Brown |
| 10 | `#B8B0AC` | Gray |

These are **deliberately desaturated** vs Tableau 9's gem-tone defaults — Stone's research showed muted hues are easier to discriminate at small sizes and reduce eye strain over long viewing.

### 9.4 Order-encoding properties

Test any sequential colormap by:

1. Render a 256-step gradient as a band.
2. Convert to grayscale (multiply linear-RGB channels by `0.2126, 0.7152, 0.0722` and threshold to 8-bit).
3. Verify: monotonic from dark to light, no plateaus, no inversions.

Viridis passes; jet fails.

## Part 10 — Color in Dark Mode Specifically

### 10.1 Saturation must DECREASE on dark backgrounds

A saturated cyan `#00FFFF` on `#FFFFFF` is unremarkable — both are bright; the eye barely registers cyan as colored. The same `#00FFFF` on `#000000` is **searingly bright** because:

1. Helmholtz-Kohlrausch boost — saturated colors appear brighter, especially blues and cyans.
2. Iris dilation — the dark surround dilates the iris, so the bright color dumps more energy onto the retina.
3. Halation — the bright pixels bloom into the dead-pixel surround.

**Rule:** when porting a palette from light to dark mode, drop OKLCH chroma by ~30%. A light-mode `oklch(0.55 0.20 250)` becomes `oklch(0.78 0.14 250)` in dark mode — *brighter L* (so it still pops against dark bg) but *lower C* (so it doesn't vibrate).

### 10.2 Apple's auto-desaturation

iOS / macOS provide system colors (`UIColor.systemBlue`, etc.) that **automatically desaturate** when dark mode activates. The spec hint: dark-mode systemBlue is `#0A84FF`, light-mode is `#007AFF` — same hue, slightly higher L, slightly lower C.

### 10.3 Material 3 elevation

Material's dark theme uses **elevation overlays** — a translucent white at varying opacity is composited over the surface based on logical "elevation":

| Elevation | Overlay opacity |
|---|---|
| dp 0 | 0% (bg = `#121212`) |
| dp 1 | 5% white |
| dp 2 | 7% |
| dp 3 | 8% |
| dp 4 | 9% |
| dp 6 | 11% |
| dp 8 | 12% |
| dp 12 | 14% |
| dp 16 | 15% |
| dp 24 | 16% |

For DEEP6, replicate with explicit OKLCH lightness steps rather than overlay opacity (more predictable for screenshots and brand renderers).

### 10.4 Text emphasis tiers

Material's dark-theme text rule (well-cited):

| Emphasis | Opacity (white on dark) | Use |
|---|---|---|
| High | 87% (`#FFFFFFDE`) | Body text, labels |
| Medium | 60% (`#FFFFFF99`) | Secondary text, captions |
| Disabled | 38% (`#FFFFFF61`) | Disabled controls |

Pure white `#FFFFFF` is reserved for graphics and icons that demand maximum punch. **Never use pure white for body text on a dark trading UI** — halation will fatigue traders within hours.

### 10.5 White-on-dark rendering artifacts

ClearType / DirectWrite / CoreText subpixel anti-aliasing assumes *dark text on light background* and can fringe colors on the inverse. Mitigations:

- Use grayscale anti-aliasing (Direct2D `D2D1_TEXT_ANTIALIAS_MODE_GRAYSCALE`) for white-on-dark.
- Increase letter-spacing 0.01–0.02em.
- Increase font weight by ~100 for the same perceived weight (e.g., Regular dark→Medium light to balance).

## Part 11 — Color Theming Patterns

### 11.1 The two-token system

```
Raw tokens          Semantic tokens           Use site
—————————           —————————                  ————————
neutral-950   ──┐
                ├─→  bg.canvas         ──→  Chart background
neutral-900   ──┘
                
green-500     ──→   color.gain          ──→  Positive P&L
red-500       ──→   color.loss          ──→  Negative P&L
blue-500      ──→   color.signal.E1     ──→  Absorption signal
```

The semantic layer lets you re-theme without touching the consumer. The raw layer lets you maintain palette integrity.

### 11.2 Three-tier system (Radix model)

```
Token        Variant          State
—————        ——————           —————
accent      .step1 ... step12  :hover, :active, :focus, :disabled
gray         .step1 ... step12  :hover, :active, :focus, :disabled
red          .step1 ... step12  ...
green        .step1 ... step12  ...
```

Radix's 12-step semantic mapping:

```
1  — App background
2  — Subtle background (page surfaces)
3  — UI element background
4  — Hovered UI element background
5  — Active / selected UI element background
6  — Subtle borders, separators
7  — UI element border, focus ring
8  — Hovered UI element border
9  — Solid backgrounds (buttons)
10 — Hovered solid backgrounds
11 — Low-contrast text
12 — High-contrast text
```

For DEEP6, adopting the Radix 12-step is pragmatic: every UI state has a defined slot, and switching themes is changing one token table.

### 11.3 Mode switching strategy

**CSS variables** (web, NinjaTrader chart object brushes via setters): cheap, instant. One `:root` block per theme.

**Full re-render** (SharpDX brush rebuild on theme change): costs ~5ms per chart but is necessary because Direct2D brushes are bound at creation. For NinjaTrader, subscribe to a theme-change event and rebuild brush dictionaries.

```csharp
// NinjaTrader theme rebuild pattern
public class ThemeBrushes
{
    public SolidColorBrush BgCanvas;
    public SolidColorBrush ColorGain;
    public SolidColorBrush ColorLoss;

    public void Rebuild(ChartControl chart, Theme theme)
    {
        BgCanvas?.Dispose();
        ColorGain?.Dispose();
        ColorLoss?.Dispose();
        BgCanvas    = new SolidColorBrush(chart.RenderTarget,
                                          ToRawColor(theme.BgCanvasHex));
        ColorGain   = new SolidColorBrush(chart.RenderTarget,
                                          ToRawColor(theme.GainHex));
        ColorLoss   = new SolidColorBrush(chart.RenderTarget,
                                          ToRawColor(theme.LossHex));
    }

    static RawColor4 ToRawColor(string hex) {
        var c = System.Drawing.ColorTranslator.FromHtml(hex);
        return new RawColor4(c.R/255f, c.G/255f, c.B/255f, c.A/255f);
    }
}
```

### 11.4 Print mode — always white background

A trader prints a screenshot for compliance, sends a chart in a deck, or pastes into a Word doc. The dark-mode chart on white paper is a halation nightmare. Provide an **explicit "print" theme** that:

- Background → `#FFFFFF`.
- Text → `#0A0A0A`.
- Saturated accents → drop chroma by 50%, raise lightness contrast to bg.
- Diverging gradient inverts: dark red ↔ light gray ↔ dark blue, instead of bright on dark.

### 11.5 High-contrast mode (Windows accessibility)

Windows users with `SystemHighContrast` enabled expect:
- Background → `SystemColors.WindowColor` (often pure black or pure white).
- Text → `SystemColors.WindowTextColor`.
- Palette → ignored; replaced with a 4-color system palette (Highlight, HighlightText, GrayText, ButtonFace).

NinjaTrader inherits Windows theme settings via WPF; the agent should **detect and gracefully degrade** to system colors when `SystemParameters.HighContrast == true`. Custom palette overrides must check this flag first.

Negative mid:      oklch(0.42 0.10 25)    #6E2B22

## Part 14 — Color Naming Conventions

### 14.1 Three naming traditions

**Semantic ("buy" / "sell"):** ties color to the operational meaning. Pro: traders understand instantly; survives re-theming. Con: tightly coupled to the trading domain.

**Generic ("positive" / "negative"):** abstracts above any specific verb. Pro: portable across contexts (P&L, score, sentiment). Con: requires cognitive translation in any specific moment.

**Financial ("gain" / "loss"):** the formal vocabulary of trading. Pro: unambiguous, matches accounting terminology; survives regulatory scrutiny. Con: doesn't generalize to non-financial UIs.

**DEEP6 recommendation:** layer the names.

```
Layer 1 (raw):       green-500, red-500
Layer 2 (financial): gain.fill, loss.fill          // primary tokens
Layer 3 (action):    buy.signal, sell.signal       // event tokens
Layer 4 (component): candle.up, candle.down        // component tokens
```

The component layer reads `--candle-up: var(--gain-fill)`. Re-skinning a component changes Layer 4 only.

### 14.2 Why naming matters

- **Translation.** Non-English locales have different finance idiom: 益 / 損 (Japanese, gain/loss), 涨 / 跌 (Chinese, rise/fall). Semantic tokens map cleanly; raw color names ("red") become wrong in Asian locales.
- **Theming.** A "red" token forces red. A "loss" token can be red in Western, green in Asian.
- **Brand.** Future white-label deployments need brand color overrides without code changes.
- **A/B testing.** Swapping `gain.fill` from green to teal across a cohort tests perception without touching components.

## Part 15 — The Asian Market Color Inversion

### 15.1 The cultural fact

In China, Japan, South Korea, Taiwan, Hong Kong: **red = up, green/black = down.** Inverse of Western convention.

The origin: Japanese candlestick traders (18th century, Honma Munehisa) drew yang (rising) candles in red — the symbolic color of vitality, luck, prosperity in East Asia. Yin (falling) candles were black ink on white paper. Modern terminals inherit this directly. China's CSI 300 ticker board on a red-flooded day means a *good* day.

### 15.2 Implementation pattern

```typescript
type Locale = 'western' | 'east_asian';

  gainFill: string;
  lossFill: string;
}

    return {
      gainFill: 'oklch(0.62 0.20 25)',   // red = up

    };
  }
  return {
    gainFill: 'oklch(0.66 0.16 145)',   // green = up
    lossFill: 'oklch(0.58 0.19 25)',    // red = down
  };
}
```

### 15.3 Detection and override

**Detection signals (in order of precedence):**
1. Explicit user setting (`settings.locale = 'east_asian'`).
2. Account region from broker (Apex US → western; Lucid HK → east_asian).
3. Browser/OS language (`zh-CN`, `ja-JP`, `ko-KR` → east_asian).
4. Geolocation (least reliable; behind VPN often wrong).

**User override:** always provide a top-level toggle. Some Asian traders working Western markets want the Western convention to match the firms they trade for; some Western expats living in Asia want the Asian convention. The setting must be one click away.

### 15.4 What stays constant

The semantics of `gain` and `loss` never change. Only the hue mapping does. Down the rest of the pipeline, every component reads `colors.gainFill` — it never knows whether that resolves to red or green.

## Part 16 — Anti-Pattern Catalog

| Anti-pattern | Why it fails | Fix |
|---|---|---|
| **Rainbow / jet colormap** | Discontinuous lightness, false features | Viridis (uniform) or Turbo (high-contrast) |
| **HSL palette generation** | Lightness varies wildly with hue | Generate in OKLCH, fixed L per step |
| **Pure black `#000` background** | OLED smear, halation, vibration with saturated accents | `#0A0A0A` to `#101418` |
| **Pure white text on dark** | Halation for astigmatic users, harsh in long sessions | `#E8EBEE` (≈87% white) |
| **Red/green only for buy/sell** | 8% of male traders cannot reliably distinguish | Add shape, position, label |
| **Saturated colors at large sizes** | Eye fatigue within 30 minutes | Tier 3 saturation only for small/transient elements |
| **Linear RGB gradient interp** | Muddy mid-tones, dead zones | Bezier in OKLCH/Lab |
| **Same chroma across all hues at fixed L** | Yellow looks too bright, blue too dim | Helmholtz-Kohlrausch compensation: drop blue C, drop red C |
| **Single-value Lc threshold for all sizes** | Small text + low contrast still passes APCA Lc 60 | Use full APCA size table; ≥18px requires Lc 75 |
| **Fixed light-mode palette inverted for dark** | Saturated colors vibrate on dark | Re-derive: higher L, lower C |
| **No print mode** | Compliance prints are illegible | Explicit white-bg theme |
| **No high-contrast detection** | Windows accessibility ignored | Check `SystemParameters.HighContrast` |
| **Hard-coded "red" hex per component** | Localization breaks (Asian markets) | Token through `gain` / `loss` |
| **Same color for two near-luminance signals** | Confusable for tritanopes | ΔE2000 ≥ 10 verified pairs |

## Part 17 — Mathematical Reference Card

### 17.1 sRGB transfer function (gamma)

```
sRGB → linear:
  if c ≤ 0.04045:  c_lin = c / 12.92
  else:            c_lin = ((c + 0.055) / 1.055)^2.4

linear → sRGB:
  if c ≤ 0.0031308: c_srgb = c * 12.92
  else:             c_srgb = 1.055 * c^(1/2.4) - 0.055
```

### 17.2 Relative luminance (WCAG)

```
Y = 0.2126·R_lin + 0.7152·G_lin + 0.0722·B_lin
```

### 17.3 WCAG contrast ratio

```
ratio = (max(Y1, Y2) + 0.05) / (min(Y1, Y2) + 0.05)
```

### 17.4 APCA Lc (BoW polarity, dark text on light bg)

```
Y_bg  = 0.2126729·R_bg^2.4  + 0.7151522·G_bg^2.4  + 0.0721750·B_bg^2.4
Y_txt = 0.2126729·R_txt^2.4 + 0.7151522·G_txt^2.4 + 0.0721750·B_txt^2.4

if Y_bg < 0.022:  Y_bg  += (0.022 - Y_bg)^1.414
if Y_txt < 0.022: Y_txt += (0.022 - Y_txt)^1.414

if |Y_bg − Y_txt| < 0.0005: return 0

if Y_bg > Y_txt:                                  // BoW
    SAPC = (Y_bg^0.56 − Y_txt^0.57) * 1.14
    Lc   = (SAPC < 0.1) ? 0 : (SAPC − 0.027) * 100
else:                                             // WoB
    SAPC = (Y_bg^0.65 − Y_txt^0.62) * 1.14
    Lc   = (SAPC > −0.1) ? 0 : (SAPC + 0.027) * 100
```

### 17.5 ΔE2000 (color difference)

The full CIEDE2000 formula is published in Sharma, Wu, Dalal 2005 (the canonical implementation reference). Too long to inline. Use `colour-science` (`colour.delta_E(lab1, lab2, method='CIE 2000')`) or chroma.js (`chroma.deltaE(c1, c2)` which uses CIEDE2000).

Interpretation:
- ΔE < 1: imperceptible
- 1 ≤ ΔE < 3: perceptible on close inspection
- 3 ≤ ΔE < 5: noticeable
- ΔE ≥ 5: clearly different
- ΔE ≥ 10: obviously different

### 17.6 OKLCH ↔ sRGB pipeline (full code)

```python
import numpy as np

def srgb_to_linear(c):
    return np.where(c <= 0.04045, c / 12.92, ((c + 0.055) / 1.055) ** 2.4)

def linear_to_srgb(c):
    return np.where(c <= 0.0031308, c * 12.92, 1.055 * c ** (1/2.4) - 0.055)

def srgb_to_oklab(rgb):
    r, g, b = srgb_to_linear(np.asarray(rgb, dtype=float))
    l = 0.4122214708*r + 0.5363325363*g + 0.0514459929*b
    m = 0.2119034982*r + 0.6806995451*g + 0.1073969566*b
    s = 0.0883024619*r + 0.2817188376*g + 0.6299787005*b
    l_, m_, s_ = np.cbrt(l), np.cbrt(m), np.cbrt(s)
    L = 0.2104542553*l_ + 0.7936177850*m_ - 0.0040720468*s_
    a = 1.9779984951*l_ - 2.4285922050*m_ + 0.4505937099*s_
    b2 = 0.0259040371*l_ + 0.7827717662*m_ - 0.8086757660*s_
    return L, a, b2

def oklab_to_srgb(L, a, b):
    l_ = L + 0.3963377774*a + 0.2158037573*b
    m_ = L - 0.1055613458*a - 0.0638541728*b
    s_ = L - 0.0894841775*a - 1.2914855480*b
    l, m, s = l_**3, m_**3, s_**3
    r = +4.0767416621*l - 3.3077115913*m + 0.2309699292*s
    g = -1.2684380046*l + 2.6097574011*m - 0.3413193965*s
    b2 = -0.0041960863*l - 0.7034186147*m + 1.7076147010*s
    return linear_to_srgb(np.array([r, g, b2]))

def oklch_to_oklab(L, C, h_deg):
    h = np.deg2rad(h_deg)
    return L, C * np.cos(h), C * np.sin(h)

def oklch_to_hex(L, C, h_deg):
    rgb = oklab_to_srgb(*oklch_to_oklab(L, C, h_deg))
    rgb = np.clip(rgb, 0, 1)
    r, g, b = (np.round(rgb * 255).astype(int))
    return f"#{r:02X}{g:02X}{b:02X}"
```

## Part 19 — Final Decision Tree for the Agent

```
Is the element BACKGROUND or CHROME?
  → Use surface.* (low chroma, neutral)

Is it TEXT?
  → Use text.primary (default), text.secondary (less important), text.muted (disabled)
  → Verify APCA |Lc| ≥ 75 vs the actual surface it sits on

Is it FINANCIAL DATA (price, P&L, position)?
  → Use gain.* / loss.* tokens (locale-aware)
  → Add shape and label, never color alone

Is it a SIGNAL?
  → Use signal.<family> for the family color
  → Differentiate within family by shape, size, fill
  → Verify CVD distinguishability vs other concurrent signals

Is it a HEATMAP?
  → Sequential ordered data: viridis (default) or inferno (high contrast)
  → Diverging around 0: delta.* gradient
  → Categorical: Okabe-Ito or signal.* family

Is it an ALERT or FLASH?
  → Use the .bright variant; saturated, brief
  → Animate decay back to base over 200-400ms

Is the user in EAST ASIAN locale?
  → Swap gain ↔ loss hue assignments
  → All other tokens unchanged

Is HIGH CONTRAST mode enabled (Windows)?
  → Defer to system colors entirely; ignore palette

Is this a PRINT export?
  → Switch to print theme (white bg, dark text)

Two emitted colors must be distinguishable?
  → Run cvd_safe(c1, c2) for protan/deutan/tritan
  → Require ΔE2000 ≥ 10 for all three
```

## Closing notes for the agent

1. **OKLCH is the source-of-truth.** Every color decision starts there; sRGB hex is the *output*, never the *spec*.
2. **Generate, then verify.** Generate the palette algorithmically, then run the CVD-safety, APCA-contrast, and ΔE2000 verification. Refuse to ship a palette that fails.
3. **Tier saturation strictly.** Background/chrome → near-zero chroma. Data → moderate chroma. Signals/alerts → high chroma but small size. The Helmholtz-Kohlrausch effect punishes saturated colors at scale.
4. **Layer naming.** Raw → semantic → component. Theming changes one layer; components untouched.
5. **Locale matters.** Western and East Asian finance invert green/red. Build the swap into the architecture; do not hard-code "green = up."
6. **Pure black is a trap.** OLED smear, halation, color vibration. Use `#101418`–`#0D1117` range.
7. **APCA over WCAG 2.** WCAG 2 contrast is symmetric and ignores polarity; APCA matches perception. Both should be reported, but **APCA Lc is the gate**.

This reference is the foundation for every color decision the agent will make. Treat the OKLCH coordinates as the spec, the hex codes as the rendered output, the CVD/APCA verifiers as the build gate, and the anti-pattern catalog as the lint rules.
