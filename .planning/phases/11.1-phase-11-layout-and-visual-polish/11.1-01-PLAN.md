  <files>dashboard/app/page.tsx, dashboard/components/score/ScoreWidget.tsx</files>

Fix the three-column flex shell so all three columns maintain their intended widths.

**Root cause (D-01):** ScoreWidget's root div in ScoreWidget.tsx has `w-60` but not `shrink-0`.
Flex default is `flex-shrink: 1`, meaning even with `w-60` set as initial width, the browser
will compress the ScoreWidget when total content width exceeds the container. Add `shrink-0`
to ScoreWidget's root div so the 240px column is non-negotiable.

In `dashboard/components/score/ScoreWidget.tsx` line ~74, change:
```tsx
<div className="w-60 bg-bg-surface p-4 flex flex-col gap-3 border-l border-border-subtle overflow-y-auto">
```
to:
```tsx
<div className="w-60 shrink-0 bg-bg-surface p-4 flex flex-col gap-3 border-l border-border-subtle overflow-y-auto">
```

**Root cause (D-02):** Without `overflow: hidden` on the flex container, any child that
internally grows wide (e.g., a canvas resizing event or an unclipped flex item) can cause
the whole row to reflow, pushing ScoreWidget off screen or causing SignalFeed to visually
overflow. Add `overflow-hidden` to the 3-column row container in page.tsx.

In `dashboard/app/page.tsx` line ~62, change:
```tsx
<div className="flex-1 flex min-h-0">
```
to:
```tsx
<div className="flex-1 flex min-h-0 overflow-hidden">
```

Additionally, confirm FootprintChart already has `flex-1 min-w-0` effective — its current
`min-w-[600px]` sets a minimum but does not provide the flex shrink floor. If at 1440px the
chart + 320 + 240 = correct, no further change needed. Do NOT add additional constraints to
FootprintChart unless the overflow-hidden fix alone is insufficient.

Both changes together ensure: SignalFeed stays inside its 320px column (overflow cannot

<task type="auto">
  <name>Task 2: Header strip typography and global font baseline (D-03)</name>
  <files>dashboard/app/globals.css, dashboard/components/layout/HeaderStrip.tsx</files>
  <action>
Fix header strip text rendering at UI-SPEC's 13px.

**Root cause (D-03):** Tailwind v4 uses `@import "tailwindcss"` which includes a modern
CSS reset (preflight) that sets `html { font-size: 100% }` (resolves to browser default 16px).
This wins over the `html, body { font-size: 13px }` rule in globals.css because the
`@import "tailwindcss"` directive injects the preflight BEFORE user rules in the cascade
when using `@import` at the top of the file. The `text-[13px]` on HeaderStrip should work
as a local override, but if the header's font-size is being cascaded from a parent reset,
the utility class must be explicit.

**Fix 1 — globals.css:** After the `:root {}` block and `@theme inline {}` block, ensure
the `html, body` font-size rule uses `!important` to override the Tailwind preflight reset,
OR reorder so the rule appears after the `@import`. Since the `@import` is at line 1, the
current placement of `html, body { font-size: 13px }` at line ~66 should already cascade
correctly (author styles after @import = higher specificity). However, add an explicit
`font-size: 13px !important` to `html` only (not body) to guarantee the base:

```css
html {
  font-size: 13px !important;
}
```

Add this rule directly after the `@theme inline {}` block and BEFORE the combined
`html, body {}` rule. Keep the existing `html, body {}` rule intact — the `!important`
on `html` alone acts as the cascade anchor.

**Fix 2 — HeaderStrip.tsx:** The `text-[13px]` class on the `<header>` element is correct
per UI-SPEC. Add `leading-none` to the header to prevent line-height inflation from
compressing the effective text size visibility. Also add `font-size: 13px` as an inline
`style` attribute as a belt-and-suspenders explicit override, which cannot be overridden
by any Tailwind class or cascade:

Change the header element's className to add `font-[13px]` and add `style={{ fontSize: '13px' }}`.

In `HeaderStrip.tsx` line ~33:
```tsx
<header
  className="h-10 bg-bg-surface border-b border-border-subtle flex items-center px-4 gap-3 text-[13px] shrink-0"
  style={{ fontSize: '13px' }}
>
```

The dual approach (CSS rule + inline style) ensures legibility regardless of which reset
wins in the cascade. Per D-03 locked decision.
